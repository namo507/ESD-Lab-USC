## Question 1: How do I envision the team using this dashboard?

The way I have been thinking about the dashboard from the start, it is meant to be the single operational pane of glass for the NANO Study, not a one off poster. Every time someone on the team opens it, the goal is that they walk away within thirty seconds knowing exactly where we stand against the R01 plan and what is blocking the next set of decisions. I built it as a continuous SPA served by `dashboard/server/live_dashboard_server.py` and mirrored on Cloudflare Pages, so the same routes work locally for the lab machine and remotely for collaborators who only need read access. The landing route at `/` opens directly on a homepage snapshot of the most important operational metrics, and the operator route at `/overview` exposes the full nine section continuous page with a jump atlas, view panel, local section chips, and a left sidebar. I added the auto and forced light or dark theme control so that the deck still presents cleanly in a bright NICU conference room or a dim office during analysis review, which I found mattered when I tested it on different monitors.

In day to day use, I see four primary patterns. First, Dr. Bradshaw and the Co-Investigators can use the Overview, ESD Lab Site, and ESD Impact tabs as a weekly leadership snapshot. The enrollment line chart, the stacked progress bar against the 65 ASIB, 130 PT, 65 TD targets, and the five KPI cards answer the recurring question of where we are versus the five year plan without anyone needing to open R or Python. Second, the Data Coordinator and the RAs can sit on the Data Quality and Cohort Table tabs during their queries call. The missingness bar, the QC flag cards, the queries per event chart, the surrogate participant table, and the REDCap audit log give them a clear list of what needs to be cleaned before the next freeze, and the small `i` hint pop ups on every KPI card spell out the exact REDCap field or pipeline step the number was pulled from, which removes ambiguity in those meetings. Third, Dr. O'Reilly and the biostatistician can rely on the Pipeline, ML Performance, and Trajectories tabs as a running validation surface. The thirteen step pipeline SVG, the architecture explainer animation, the ROC curves with AUROC and confidence intervals, the SHAP plots, the subgroup sensitivity panel, and the trajectory plot with confidence bands and intercept and slope tables let them sanity check a new model run side by side with the previous one without me having to push static figures around. Fourth, the Reading Library tab is a quiet but heavily used surface for everyone, because it auto indexes any PDF dropped into `ESD Lab readings/`, and the team gets a searchable, category chipped view of the most recent literature instead of an email thread.

For the team presentation, the short bullets I would put on the slide are these. The dashboard is a live, HIPAA safe operational view of the NANO Study that answers four questions: where we are against the R01 plan, whether the data are clean enough to analyze, whether the models are any good and how they work, and whether the groups are diverging over time. It is the same surface for the leadership snapshot, the weekly queries call, the model validation review, and the literature scan, which means no one has to maintain a separate slide deck or spreadsheet for any of those. It strips PHI before anything reaches the browser, so it can be shared with the wider lab and with external collaborators on the Cloudflare Pages link without extra IRB friction. It updates automatically when the nightly REDCap sync and the build pipeline run, which means whatever someone sees at 9 am is what the latest pipeline produced, not a stale screenshot. And it has a jump atlas, a view panel, local chips, and a sidebar so each role on the team can land on the panel they actually need within a click or two.

## Question 2: Is the dashboard currently on simulated data, have I worked with REDCap before, and can we wire it to MATLAB?

Yes, you are right that right now the dashboard is running on simulated data. The payload it loads, `dashboard/data/dashboard_data.json`, currently reports a `data_source` of `repo_demo_inputs`, which is the synthetic demo path that `dashboard/pipelines/generate_synthetic_dashboard_data.py` produces. I built the dashboard intentionally so that when the secure data mount is unavailable it falls back to synthetic data, which is why the UI and the readings library still render cleanly while we are in the demo phase. The numbers you see for ASIB at 52 of 65, PT at 105 of 130, and TD at 53 of 65, along with the HRV trajectories, ROC curves, and missingness bars, are all generated by that demo script using the same schema the production pipeline emits. They are realistic by construction, but they are not real participant data. The schema itself, documented in `dashboard/context_skill/references/dashboard_schema.md`, is the contract that ties the demo and the production paths together, so the moment we point the runtime at a live REDCap pull and the processed feature parquet, the dashboard will swap from synthetic to live without any UI change. There is even a `data_source` chip in the top bar that will then read something other than `synthetic_demo`, which is the first thing I always tell the team to check if a number looks off.

On the REDCap question, yes I have worked with REDCap before and the production code path is already written. The repository ships with `redcap/api/redcap_pull.py`, `redcap/api/redcap_push.py`, `redcap/api/redcap_audit.py`, and an R sibling at `redcap/api/redcap_r_pull.R`. The Python module uses the `PyCap` library and reads `REDCAP_API_TOKEN` and `REDCAP_API_URL` from a local `.env` file that is never committed. It instantiates an authenticated `redcap.Project`, pulls all longitudinal records with paginated chunked requests, supports an event filter so we can refresh just `month_12_arm_1` or any single event, and writes the results outside the repo into the secure data root. The setup guide at `docs/redcap_setup_guide.md` walks through the longitudinal project, the arms and events, the instruments, the API token, the external module deployment for JS hooks, the automated survey invitations, and the double data entry configuration. There are also four JS hooks under `redcap/hooks/` for auto completing the date of birth, flagging missing ECG sessions, validating the participant ID format, and checking visit completion. The nightly cron at `scripts/redcap_daily_sync.py` is what ties everything together once your team issues an API token. The moment we have the token and the secure mount, I can flip the build pipeline from synthetic to live in a single config change and the dashboard will start rendering real numbers.

On connecting to MATLAB, yes there are several clean ways to do it and the choice depends on which direction the data are flowing and how often. The cleanest pattern, and the one I would recommend for the large, densely sampled signals that your team manages through MATLAB scripts, is to treat MATLAB as another upstream producer in the same way we treat REDCap. The MATLAB scripts already run on the secure server, so we can have them write their derived feature tables to the shared `data/interim/` or `data/processed/` location as Parquet or CSV files, and the existing Python pipeline at `dashboard/pipelines/build_dashboard_data.py` will merge them into the feature matrix using the NANO-XXXX surrogate ID and the visit event key just like it merges the ECG and REDCap streams today. MATLAB has had a native Parquet reader and writer since R2022a, and it can write CSV with one call, so no glue code is needed on the MATLAB side beyond an export line at the end of each script. That keeps the dashboard agnostic to the source language and avoids a tight runtime coupling. For the REDCap to MATLAB direction, MATLAB can call the REDCap API directly using `webread` and `webwrite` with the same `record/` endpoint and the same API token we use in Python, so the densely sampled scripts can pull the visit metadata they need without going through the Python layer. If at some point we want true interop, we can stand up the MATLAB Engine API for Python and call a `.m` function from inside `build_dashboard_data.py`, or expose the MATLAB analysis as a tiny REST endpoint using MATLAB Production Server or a Flask wrapper around `matlab.engine`, which the dashboard could then query just like any other data source. For most of what your team is doing today, the Parquet handoff pattern is the lowest cost and the most maintainable, and we can layer the engine or the REST option on later if a real time analytic ever needs to be triggered from a click in the UI.
